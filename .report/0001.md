# Инкремент 0001 — Анализ возможности запуска NetBSD 11.0-RC2 на JPC

**Дата:** 2026-04-30
**Статус:** Аналитика и план (кода в этом инкременте нет).
**Целевая ОС:** NetBSD 11.0-RC2 (выбран target **amd64**; см. §5 — почему рекомендуется откатиться к **i386**).

---

## 1. Архитектура эмулируемого ПК (текущий JPC)

### 1.1. CPU

Чисто 32-битный x86, реализован программно. Источники: `jpc-core/src/main/java/org/jpc/emulator/processor/Processor.java`, `jpc-core/src/main/java/org/jpc/emulator/execution/`.

| Свойство                | Значение                                                                                  |
|-------------------------|-------------------------------------------------------------------------------------------|
| Режимы                  | real, protected, virtual-8086 (`opcodes/{rm,pm,vm}/`)                                     |
| Long mode (IA-32e/x64)  | **Отсутствует** — ни кода, ни упоминаний `EFER`, `LME`, `PML4`, `REX`, `RIP`              |
| Уровни CPU              | 486 (`cpuLevel=4`), Pentium (`5`), Pentium II (`6`) — выбирается через `-cpulevel`        |
| CPUID feature flags (PII)| PAE, SYSENTER/SYSEXIT, CMOV, MMX, FPU, TSC, MSR, PSE, DE, VME (см. `Processor.cpuid` ~L1336) |
| SSE / SSE2              | **Не объявлены в CPUID** (бит CR4.OSFXSR определён константой, но реальных SSE-опкодов нет) |
| FPU                     | Есть, отдельный пакет `processor/fpu64/`                                                  |
| MSR                     | Минимальный набор — `SYSENTER_*`, EFER не реализован                                      |
| APIC / Local APIC       | Нет (`grep -r APIC` пустой по эмулятору)                                                  |
| SMP                     | Один логический CPU; нет MP-table, нет AP-startup IPIs                                    |
| Декодер                 | Генерируется из `jpc-tools/src/main/resources/Opcodes_*.xml` (3 таблицы по режимам)        |
| Кодогенерация           | `Interpreted{Real,ProtectedMode,VM86Mode}Block` + `OptimisedCompiler` + `BackgroundCompiler`|

### 1.2. Память

- Источник: `jpc-core/src/main/java/org/jpc/emulator/PC.java:71-72`, `memory/PhysicalAddressSpace.java`.
- `SYS_RAM_SIZE` — статический `int`. По умолчанию `Option.ram` ⇒ `16 МБ`.
- Адреса в `PhysicalAddressSpace` маскируются `& 0xffffffffL` — теоретический потолок ~4 ГБ, но на практике не тестировалось; типовые сценарии — десятки МБ.
- Страничная таблица: 32-битная, PAE-флаг в CPUID есть, но **полноценный PAE walk** в `LinearAddressTranslator` нужно перепроверить (см. §3.4).

### 1.3. Чипсет / motherboard

`jpc-core/src/main/java/org/jpc/emulator/motherboard/`, `pci/`:

- BIOS: `resources/bios/bios.bin` (128 КБ, Bochs-style; класс `SystemBIOS` парсит сообщения Bochs BIOS).
- VGABIOS: `resources/bios/vgabios.bin`.
- PIT: `BochsPIT` (порт из Bochs).
- PIC, RTC, DMA, IO-port routing, A20 gate — присутствуют.
- PCI-host: `PCIBus`; видеокарта — `DefaultVGACard` / `VGACard` с Bochs VBE на vendor `0x1234`.
- IDE: `PIIX3IDEInterface` + `IDEChannel` — поддержка ATA и **ATAPI** (включая `WIN_PACKETCMD 0xA0`, `ATAPI_PACKET_SIZE`, идентификация ATAPI-устройств) — CD-ROM исполнимы.
- ACPI: **не реализован** (в `EPROMMemory.java` есть TODO «disabled until I understand the interplay of this with ACPI tables…»).

### 1.4. Периферия

| Категория  | Что есть                                                          | Чего нет                       |
|------------|-------------------------------------------------------------------|--------------------------------|
| Диск       | IDE/ATA + ATAPI (CD-ROM); Floppy                                  | AHCI/SATA, NVMe, virtio-blk, SCSI |
| Сеть       | NE2000 (Bochs port, `EthernetCard`)                                | e1000, rtl8139, virtio-net     |
| USB        | —                                                                  | UHCI/EHCI/xHCI, HID, mass-storage |
| Видео      | Bochs VGA + VBE (cirrus-подобное)                                  | qxl, virtio-gpu                |
| Звук       | SB16, AdLib, MPU-401, PC speaker                                   | HDA/AC97 (часто не нужно)      |
| Ввод       | PS/2 keyboard + mouse                                              | USB HID                        |
| Serial     | 4 порта (16550-style)                                              | —                              |

### 1.5. Boot

- `DriveSet.BootType = {FLOPPY, HARD_DRIVE, CDROM}` — есть **bootable CD-ROM** через El Torito (Bochs BIOS поддерживает).
- Манифест по умолчанию: `-fda mem:resources/images/floppy.img -hda mem:resources/images/dosgames.img -boot fda` (FreeDOS).
- UEFI: **не поддерживается**.

### 1.6. Что грузится сегодня

DOS, Windows 9x, ряд Linux. Это ровно тот класс гостей, для которых Pentium II + ATA + VGA + NE2000 + Bochs BIOS — достаточная платформа.

## 2. Минимальные требования NetBSD 11.0-RC2

### 2.1. NetBSD/i386 11.0-RC2 (для сравнения)

Источник: [INSTALL.html для i386 11.0_RC2](https://test.nic.funet.fi/pub/OS/NetBSD/NetBSD-11.0_RC2/i386/INSTALL.html), [supported hardware](https://www.netbsd.org/ports/i386/hardware.html).

- **CPU:** i486 или новее, **обязателен x87 FPU**.
- **RAM:** 64 МБ для стандартной установки (32 МБ — кастомный stripped kernel вручную).
- **Диск:** 2 ГБ для полной установки.
- **Boot:** CD/DVD/USB или комплект floppy-образов (`boot1.fs`+`boot2.fs` или `boot-com*` для serial).
- **SMP:** 1–32 CPU.

### 2.2. NetBSD/amd64 11.0-RC2 (выбранный target)

Источник: [INSTALL.html для amd64 11.0_RC2](https://test.nic.funet.fi/pub/OS/NetBSD/NetBSD-11.0_RC2/amd64/INSTALL.html), [wiki/amd64](https://wiki.netbsd.org/ports/amd64/).

- **CPU:** AMD Opteron / Athlon64 / Intel EM64T или новее ⇒ **x86-64 long mode**.
- **Подразумевается ABI-минимум:** SSE2, FXSR, CMOV, CMPXCHG8B, CMPXCHG16B (стандарт AMD64 ABI; всё это часть базовой ISA x86-64).
- **Диск:** ≥700 МБ для базы, +225 МБ для X.
- **RAM:** в INSTALL прямо не указана; на практике GENERIC-ядро amd64 требует **≥128 МБ для установки** (исторически — `MICROVM` ядро легче, но в release notes 11.0 оно появляется как новинка).
- **Boot:** `boot.iso` (CD), `boot-com.iso` (serial), USB-образ, FTP/NFS. UEFI поддерживается, но также есть legacy BIOS-загрузчик.
- ACPI / MP-table: **обязательны** для дискаверки CPU/IRQ на amd64-ядре GENERIC.

## 3. GAP-анализ: чего не хватает JPC для NetBSD/amd64

### 3.1. Long mode (BLOCKER)

Самое крупное отсутствие. Чтобы JPC мог хотя бы стартовать amd64-ядро, нужно:

1. **Регистровый файл 64-бит:** RAX/RBX/.../R8–R15, RIP, RFLAGS-64, расширенные сегментные base/limit (FS.base, GS.base — 64-бит, через MSR).
2. **REX-префиксы** в декодере (`Prefices.java`, `OpcodeDecoder.java`, XML-таблицы).
3. **Новый декодер для long mode** — нужно добавить `Opcodes_64.xml` и сгенерировать `opcodes/lm/`.
4. **EFER MSR** (`0xC0000080`): биты SCE, LME, LMA, NXE; CR4.PAE+CR0.PG переключение в long mode.
5. **4-уровневое страничное преобразование (PML4)** в `LinearAddressTranslator`/`TLB`.
6. **64-битная сегментация** (CS.L, упрощённая модель — большинство сегментов flat).
7. **SYSCALL/SYSRET** (заменяют SYSENTER в long mode, обязательны).
8. **Прерывания/исключения 64-бит:** новый формат IDT-gate, IST в TSS, 16-байтные дескрипторы.
9. **Все опкоды должны иметь long-mode варианты:** ~400+ инструкций; включает `MOVSXD`, операции с операндами 64-бит, RIP-relative addressing, новые формы PUSH/POP, изменение значения по умолчанию для разрядности.

**Оценка трудоёмкости:** недели–месяцы. Это сопоставимо с написанием ~50% всего CPU-эмулятора заново. В контексте «минимального инкремента» — невыполнимо.

### 3.2. SSE / SSE2 / FXSR (BLOCKER)

NetBSD/amd64 пользовательское пространство компилируется с `-msse2` по умолчанию (это часть AMD64 psABI). Ядро тоже использует SSE2 в criticial paths (memcpy, etc.).

Нужно: XMM0–XMM15, состояние MXCSR, `FXSAVE`/`FXRSTOR`, CR4.OSFXSR/OSXMMEXCPT, инструкции SSE/SSE2 (≈140 опкодов), CPUID feature bits 25 (SSE) и 26 (SSE2).

В JPC сейчас бит CR4.OSFXSR определён как константа (`Processor.java:80`), но самого FXSAVE нет, и SSE-опкоды отсутствуют в XML.

### 3.3. ACPI / MP-Table

NetBSD/amd64 GENERIC при отсутствии и MP-table, и ACPI RSDP в нижних 1 МБ, panic'нет на этапе CPU discovery либо упадёт в degenerate UP-конфиг с большими шансами зависнуть на IRQ-routing.

Минимум: статический MP-table (Intel MultiProcessor Specification 1.4) для 1 CPU + IO-APIC. ACPI — желательно (RSDP/RSDT/MADT/FADT/DSDT-stub).

### 3.4. PAE и большой объём RAM

Даже если откатиться к i386, поверх 16 МБ умолчанию надо явно поднимать `-ram 256` и убедиться, что:
- `PhysicalAddressSpace` корректно отдаёт страницы при размерах ≫ 16 МБ;
- `LazyCodeBlockMemory` не упирается в ассумпции;
- E820/INT 15h AX=E820h в Bochs BIOS отдаёт корректный memory map для большого RAM (через CMOS-регистры, как в QEMU/Bochs).

### 3.5. APIC / Local APIC / IO-APIC

amd64 GENERIC ожидает Local APIC даже на UP. Без LAPIC ядро не сможет корректно настроить таймер и MSI; некоторые версии способны падать назад на 8259+PIT, но это всё реже.

### 3.6. Long-running concerns

- **HPET** (часто требуется на современных ядрах) — отсутствует.
- **AHCI/virtio-blk** — не нужны при ATAPI/ATA, но ATA emulator должен переживать тяжёлые I/O последовательности (NetBSD `wd(4)` подаёт DMA-нагрузку).
- **PCI BIOS / MCFG** — для ECAM-конфигурации; legacy CF8/CFC у NetBSD работает, так что вероятно OK.

## 5. Рекомендация: переключить target на NetBSD/i386

Аргументы:

1. **Пропасть огромная.** Long mode + SSE2 + ACPI/APIC — это ~6 крупных подсистем CPU/чипсета, каждая равна по объёму нескольким месяцам работы. JPC за ~25 лет существования long mode не получил — это не случайность, его авторы сознательно остались на 32-бит как на «истории x86».
2. **На i386 пропасть умеренная.** Платформа уже имеет PII-CPU, ATAPI CD-ROM, NE2000, FPU. Не хватает в основном LAPIC stub + MP/ACPI. Это 2–4 инкремента.
3. **i386 NetBSD 11.0 актуален.** Это полноценный, поддерживаемый порт; никакой «второсортности» по сравнению с amd64 в плане задачи «boot до sysinst».
4. **Сохранение цели «time-travel и debugger».** Time-travel в JPC — это его killer feature. Расширение CPU на long mode потребует переписать `ProcessorAccess`/`TimeTravelProcessorAccess` — большой риск регрессии time-travel.

### Если всё же amd64 — обязательные предварительные решения

- Готовы ли мы потратить ≥3–6 месяцев и принять, что time-travel debugger может временно сломаться?
- Готовы ли мы регенерировать декодер с нуля (XML-таблицы вырастут в ~1.5 раза, кодгенерация может упереться в Java method size limit 64 КБ — придётся разбивать на под-классы)?

## 7. Предложения по улучшению

Сюда складываю всё, что заметил при анализе, но что выходит за рамки задачи. Эти пункты можно вынести в отдельные инкременты или тикеты.

- **`SYS_RAM_SIZE` — статический `int` поля `PC.java:71`.** Это запрещает корректно поднять две `PC` в одном JVM с разным RAM. Кандидат на инстанс-поле.
- **Default RAM = 16 МБ** — слишком мало для всех современных ОС. Стоит поднять до 64–128 МБ, либо хотя бы вывести предупреждение в `Option.printHelp` если `-ram` не задан и `-hda`/`-cdrom` указывает на ISO/образ ≥1 ГБ.
- **`EPROMMemory.java`** — два TODO про ACPI-таблицы и processor identification, которые «ломают QEMU Linux». Стоит разобраться: возможно, тот же вопрос всплывёт при попытке загрузить любую современную ОС.
- **`CPUID` features** — комментарии вида `// SYSENTER/SYSEXIT` уже включены в PII-ветку, но в Pentium-ветке `(1<<11)` закомментирован. Hot path — выяснить, разрешить ли SYSENTER на Pentium (он там физически появился у Pentium Pro, но Bochs объявляет на Pentium-class иногда).
- **Один-единственный вид `EthernetCard`** — нет переключателя на e1000/rtl8139. Для NetBSD это OK (ne2000 поддерживается), но для будущего стоит абстрагировать.
- **`SpanningDecodeException.java~` в git status** — судя по `git status`, файл-«тильда» от редактора попал в индекс (строка `R  src/...SpanningDecodeException.java~ -> jpc-core/...SpanningDecodeException.java~`). Это явно случайно — стоит удалить из репо в отдельном housekeeping-коммите.
- **Boot UX:** при попытке загрузить ОС, не помещающуюся в 16 МБ default, эмулятор падает с малопонятной ошибкой. Желательно — early check в `JPCApplication` с подсказкой `увеличьте -ram`.
- **CLI options централизованы в `Option.java`** (как и сказано в CLAUDE.md), но help-текст вручную поддерживается в `printHelp()`. Дрейф. Кандидат на автогенерацию help из аннотаций / поля `Opt.description`.
