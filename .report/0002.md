# Инкремент 0002 — B1+B2: PII по умолчанию + MSR-stubs

**Дата:** 2026-04-30
**Этап:** B (i386-совместимость с NetBSD), подэтапы B1 + B2.
**Статус:** Реализация.
**Зависит от:** [0001.md](0001.md) — анализ.

---

## 1. Цель

Подготовить фундамент для дальнейших инкрементов B-серии (LAPIC stub, MP-table, ACPI), сделав так, чтобы:

1. По умолчанию JPC рапортует через CPUID **Pentium II**, а не Pentium. NetBSD/i386 11.0 формально требует «Pentium-compatible or later», и многие современные ОС лучше реагируют на CPUID семейства 6 (PII) с PAE/SYSENTER/CMOV/MMX в feature flags.
2. Чтение системных MSR, к которым обращается NetBSD/современные Linux на boot, возвращает **разумные значения**, а не нули. Это исключает целый класс «странных» падений при инициализации SMP/MTRR/PAT даже до того, как мы сделаем настоящий LAPIC.

Минимальный инкремент: код + JUnit-тесты + зелёная сборка. **Без реального запуска NetBSD ISO** (это уже B7).

## 2. Объём работ

### 2.1. B1 — default cpuLevel = 6

| Файл                                                               | Изменение                                              |
|--------------------------------------------------------------------|--------------------------------------------------------|
| `jpc-core/src/main/java/org/jpc/emulator/processor/Processor.java` | Строка 167: `intValue(5)` → `intValue(6)`              |
| `jpc-core/src/main/java/org/jpc/emulator/processor/Processor.java` | Строка ~3927 (в `Processor.reset()`): `intValue(5)` → `intValue(6)` |
| `jpc-core/src/main/java/org/jpc/j2se/Option.java`                  | Строка 119 — обновить help-текст: «default: 6 = Pentium II» |

Совместимость:
- При явно заданном `-cpulevel 5` поведение не меняется (тесты должны это проверить).
- При отсутствии флага раньше был Pentium (5), теперь PII (6). Это **изменение default user-visible поведения** для существующих сценариев. FreeDOS/DOS-игры из штатного образа на этом не падают (Pentium II — строгий superset Pentium), но пометить как «risk» (см. §6).

### 2.2. B2 — MSR-stubs

Семь новых **public** констант индексов MSR в `Processor.java` (рядом с уже существующими `SYSENTER_*_MSR`):

```java
public static final int IA32_TSC                = 0x10;
public static final int IA32_APIC_BASE_MSR      = 0x1B;
public static final int IA32_BIOS_SIGN_ID_MSR   = 0x8B;
public static final int IA32_MTRRCAP_MSR        = 0xFE;
public static final int IA32_MISC_ENABLE_MSR    = 0x1A0;
public static final int IA32_PAT_MSR            = 0x277;
public static final int IA32_MTRR_DEF_TYPE_MSR  = 0x2FF;
```

Новый приватный метод `seedDefaultMSRs()`:

| MSR                       | Default      | Что это                                                              |
|---------------------------|--------------|----------------------------------------------------------------------|
| `IA32_APIC_BASE` (0x1B)   | `0xFEE00900` | LAPIC base = `0xFEE00000`, BSP=1 (bit 8), Global enable=1 (bit 11). Без этого OS видит «LAPIC выключен и его адрес 0», что хуже, чем «LAPIC заявлен, но MMIO ещё не отвечает». |
| `IA32_MTRRCAP` (0xFE)     | `0x508`      | VCNT=8 (8 переменных диапазонов), WC=1 (бит 10), FIX=1 (бит 8). NetBSD при чтении видит «MTRR имеется, но мы не пользуемся». |
| `IA32_MTRR_DEF_TYPE` (0x2FF) | `0`        | MTRR globally disabled, default type=UC. Безопасный нейтральный.     |
| `IA32_PAT` (0x277)        | `0x0007040600070406L` | Стандартный Intel reset value: WB, WT, UC-, UC, WB, WT, UC-, UC. |
| `IA32_MISC_ENABLE` (0x1A0)| `0x1`        | Fast Strings enable. Минимально ожидаемое OS значение.               |
| `IA32_TSC` (0x10)         | (не сидим)   | Уже корректно возвращается через RDTSC; для RDMSR оставляем 0 → fallback в `getMSR`. |
| `IA32_BIOS_SIGN_ID` (0x8B)| (не сидим)   | NetBSD читает чтобы пропустить microcode update. 0 = no microcode loaded — корректно. |

`getMSR(int index)` уже толерантен к отсутствующим MSR (возвращает 0 + INFO-лог), так что переинженерить fallback не нужно.

### 2.3. Точки вставки

```
Processor#Processor(Clock):
    modelSpecificRegisters = new HashMap<>();
+   seedDefaultMSRs();          // ← новое

Processor#reset():
    modelSpecificRegisters.clear();
+   seedDefaultMSRs();          // ← новое (заменяет существующий комментарий «Will need to set any MSRs here»)
```

Важно: `reset()` использует `linearMemory.reset()` etc.; в JUnit мы не зовём `reset()` (он требует физпамять). Зато конструктор сидирует MSR и так — этого достаточно для регрессионных тестов.

## 4. Что НЕ входит в инкремент

- LAPIC MMIO model на 0xFEE00000 (это B3, отдельный инкремент). Сейчас просто говорим «LAPIC есть и тут адрес», но MMIO пока не отвечает.
- MP-table (B4).
- ACPI таблицы (B5).
- Реальный boot NetBSD ISO (B6/B7).
- Настоящие MTRR (адресные диапазоны не enforced; это чисто кэш-семантика).
- Default RAM bump (16 → 64 МБ) — отложен до 0003, чтобы не смешивать темы.
- Чистка `SpanningDecodeException.java~` — housekeeping в отдельном инкременте.

## 6. Риски

- **Изменение CPUID-default может сломать гостевую ОС, которая делает CPUID-вендор-проверку.** Митигация: оставлен флаг `-cpulevel 5` для отката. Штатные FreeDOS/Win9x не зависят от этого.
- **`cpuLevel` остаётся `static final` — глобален на JVM.** Для тестов это значит: если какой-то предыдущий код в этой же JVM выставил `-Dcpulevel=5`, наши тесты упадут. В обычном `mvn test` системные свойства не задаются, так что тесты пройдут. Сделать `cpuLevel` инстансным — отдельная рефакторная задача (см. §7 в 0001.md).
- **Surefire может не подхватить JUnit 5 без `surefire-junit-platform`** в новых сборках. На 3.5.0 это работает out-of-the-box (Jupiter Engine идёт транзитивно с `junit-jupiter`).

## 7. Предложения по улучшению

- **Унифицировать чтение `cpulevel` в одном месте.** Сейчас оно читается дважды: `Processor.cpuLevel` (строка 167) и локальная `cpulevel` в `reset()` (строка 3927). Выделить единый `static final int CPU_LEVEL` — устранит риск рассинхронизации.
- **`Processor.cpuLevel` как `static final`** мешает мульти-инстансной эмуляции (несколько `PC` с разным CPU-level в одной JVM). Кандидат на инстанс-поле, инициализируемое в конструкторе.
- **`seedDefaultMSRs()` стоит расширять** по мере того, как обнаруживаем новые MSR, читаемые гостями. Хорошее место для логирования с уровнем `FINE` всех неизвестных MSR-доступов, чтобы потом находить их по логу.
- **Логирование `Reading unset MSR` на уровне INFO** засоряет вывод при boot современной ОС. Имеет смысл понизить до `FINE`.
- **`-cpulevel` help-text опечатка**: «6 = Pentium Pro» — на самом деле в коде модель PII (`r_edx.set32(0x00000634)`), и features (PAE, SYSENTER, MMX, CMOV) — Pentium II. Поправим.

## 8. Источники

- Intel SDM Vol.4, MSR таблицы (IA32_APIC_BASE, IA32_PAT, MTRR).
- AMD APM Vol.2 §16.2 (Local APIC base MSR layout).
