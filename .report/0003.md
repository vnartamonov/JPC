# Инкремент 0003 — B4: Intel MP-table в BIOS shadow ROM

**Дата:** 2026-04-30
**Этап:** B (i386-совместимость с NetBSD), подэтап B4.
**Статус:** Реализация.
**Зависит от:** [0001.md](0001.md), [0002.md](0002.md).

---

## 1. Цель

NetBSD/i386 11.0 GENERIC и Linux при boot сканируют 16-байтные стрyктуры **MP Floating Pointer** (`_MP_`) в трёх ограниченных диапазонах:

1. Первая 1 KB EBDA;
2. Последняя 1 KB базовой памяти (640 K – 1 K);
3. **BIOS ROM 0xF0000–0xFFFFF.**

Если структура найдена — ОС идёт по указателю в **MP Configuration Table** (`PCMP`), узнаёт топологию (CPU, шины, IO-APIC, IRQ routing) и инициализируется по этой схеме. Если ничего не найдено и ACPI тоже отсутствует — ядро откатывается на legacy 8259 + PIT. Этот fallback в принципе работает, но (а) на современных ядрах путь хуже отлажен, (б) NetBSD печатает «MP table not found», что мешает диагностике.

**Цель инкремента:** разместить корректную статическую MP 1.4-таблицу в BIOS shadow ROM и убедиться, что её структура валидна (правильные сигнатуры и контрольные суммы).

Минимальный инкремент: код + JUnit-тесты + зелёная сборка. **Без реального запуска NetBSD ISO** (B7).

## 2. Размещение

Решение принято в уточняющих вопросах: **только BIOS shadow ROM 0xF0000–0xFFFFF**.

Конкретный 4 КБ-блок: **0xFC000–0xFCFFF**.

| Адрес            | Что лежит                     | Размер  |
|------------------|-------------------------------|---------|
| 0xFC000–0xFC00F  | MP Floating Pointer (MPFPS)   | 16 байт |
| 0xFC010–0xFC0EF  | MP Configuration Table (MPC)  | 224 байт |
| 0xFC0F0–0xFCFFF  | оригинальные байты Bochs BIOS  | 3856 байт |

**Почему 0xFC000:** 4 КБ-выровнен (`mapMemory` это требует), внутри обязательного диапазона 0xF0000-0xFFFFF, и далеко от reset-vector (0xFFFF0). У Bochs BIOS концентрация код/данных к концу образа; верх 4 КБ перед последним блоком даёт минимум коллизий.

**Почему overlay, а не патч imageData:** мы не хотим хирургически менять `byte[]` чужого `Bios`, а конструктор `EPROMMemory.copyArrayIntoContents` уже умеет принять произвольный массив. Кроме того, `addressSpace.copyContentsIntoArray()` после загрузки BIOS даёт нам реальные оригинальные байты — мы их сохраняем в xвосте 4 КБ, теряя только первые 240 байт BIOS-блока.

## 3. Объём работ

### 3.1. Новый компонент

`jpc-core/src/main/java/org/jpc/emulator/motherboard/MpTable.java` — `extends AbstractHardwareComponent`. Ключевые точки:

- Конструктор: формирует 240 байт MP-table структуры (MPFPS + MPC), кэширует в private `byte[] mpBytes`.
- `acceptComponent(PhysicalAddressSpace)`: при первом вызове после загрузки BIOS — читает оригинальный блок 0xFC000-0xFCFFF, накладывает `mpBytes` поверх первых 240 байт, оборачивает в `EPROMMemory` и вызывает `addressSpace.mapMemory(0xFC000, ...)`. Затем `loaded = true`.
- `initialised()`: возвращает `loaded`.
- Идемпотентность: повторный `acceptComponent(PhysicalAddressSpace)` пропускается через флаг.

### 3.2. Содержимое таблицы

**MPFPS (16 байт), 16-байт выровнен на 0xFC000:**

| Off | Поле                 | Значение                         |
|-----|----------------------|----------------------------------|
| 0   | Signature            | `_MP_` (0x5F 0x4D 0x50 0x5F)     |
| 4   | Phys addr of MPC     | `0xFC010` (LE)                   |
| 8   | Length / 16          | `0x01`                           |
| 9   | Spec rev             | `0x04` (1.4)                     |
| 10  | Checksum             | вычисляется (∑ % 256 == 0)       |
| 11  | Feature byte 1       | `0x00` (MPC table present)       |
| 12-15 | Feature bytes 2-5  | `0x00 0x00 0x00 0x00`            |

**MPC header (44 байт):**

| Off | Поле                 | Значение                         |
|-----|----------------------|----------------------------------|
| 0   | Signature            | `PCMP`                           |
| 4   | Base table length    | 224 (LE16)                       |
| 6   | Spec rev             | `0x04`                           |
| 7   | Checksum             | вычисляется                      |
| 8-15  | OEM ID            | `"JPC     "` (8 ASCII)           |
| 16-27 | Product ID         | `"JPC i386    "` (12 ASCII)      |
| 28-31 | OEM table ptr     | `0`                              |
| 32-33 | OEM table size    | `0` (LE16)                       |
| 34-35 | Entry count       | `21` (LE16)                      |
| 36-39 | Local APIC addr   | `0xFEE00000` (LE32)              |
| 40-41 | Ext table length  | `0`                              |
| 42  | Ext table checksum   | `0`                              |
| 43  | Reserved             | `0`                              |

**Entries (180 байт, 21 запись):**

| Кол-во | Тип               | Размер | Описание                                              |
|--------|-------------------|--------|-------------------------------------------------------|
| 1      | Processor (0)     | 20     | LAPIC ID=0, ver=0x14, flags=0x03 (enabled+BSP), CPUSig=0x634 (PII) |
| 1      | Bus (1)           | 8      | Bus ID=0, "ISA   "                                    |
| 1      | IO APIC (2)       | 8      | ID=2, ver=0x11, flags=0x01, addr=0xFEC00000           |
| 16     | IO Interrupt (3)  | 8 each | ISA IRQ 0..15 → IO-APIC INTIN. IRQ0→INTIN2, IRQ1..15→INTIN1..15 (IRQ2 пропущен — каскад) |
| 2      | Local Interrupt(4)| 8 each | LINT0=ExtINT (PIC), LINT1=NMI на all CPUs             |

**Итого тело базовой таблицы:** 44 + 20 + 8 + 8 + 16·8 + 2·8 = 224 байта. ⇒ MPC base table length = 224.

### 3.3. Подключение

`PC.java`: после `parts.add(new SystemBIOS(...))` (строка 176) добавить `parts.add(new MpTable());`. Порядок гарантирует, что в одной итерации `configure()` SystemBIOS=outer обработается раньше MpTable=outer ⇒ к моменту нашего `acceptComponent(PhysicalAddressSpace)` BIOS уже отображён.

## 5. Что НЕ входит

- LAPIC MMIO model (B3) — MP-table только декларирует адрес `0xFEE00000`, но MMIO ещё не отвечает.
- IO-APIC MMIO model — аналогично, объявлено `0xFEC00000`, MMIO нет.
- ACPI таблицы (B5) — отдельный инкремент.
- Реальный boot NetBSD (B6/B7).
- Динамическая адаптация под `-cpulevel` — entry CPU signature всегда `0x634` (PII). Если ОС читает только сигнатуру для сверки, это корректно для default `cpuLevel=6`. Для `cpuLevel=4/5` будет несоответствие, но это можно покрыть отдельно.

## 6. Риски

- **Overlay рушит первые 240 байт оригинального BIOS на 0xFC000.** Bochs BIOS концентрирует код/таблицы ближе к reset vector (0xFFFF0); 0xFC000 — менее активная область. Но 100% гарантии нет. Митигация: оставить `-bios` как способ откатить на оригинал; в случае проблемы пересоздать MpTable в другом блоке.
- **`bios.bin` уже содержит строки `_MP_`/`PCMP`** на физ. адресе 0xE1F70/0xE1F50 (вне поискового диапазона ОС). Это data templates, по-видимому не используемые рантайм Bochs BIOS в JPC. Наша таблица их не трогает.
- **Если кто-то заменит BIOS через `-bios`** на образ другого размера (≠128 КБ), мэппинг физ-адресов будет другой. Bochs BIOS-style adresses 0xFC000 могут не быть в shadow region. Митигация: если не 128 КБ — лог-WARNING и skip overlay.
- **Component init order** (см. §3.3) — если кто-то изменит порядок добавления в parts, MpTable может прочитать пустой 4 КБ блок. Тест этого не ловит — нужен интеграционный с реальным `PC`. Оставлено на будущее.
- **Тест проверяет байтовую структуру, не реальную работу с гостем.** Boot-тест с NetBSD — отдельный инкремент.

## 8. Предложения по улучшению

- **Параметризовать `cpuLevel` в Processor entry.** Сейчас CPUSig жёстко 0x634 (PII). Если default cpuLevel меняется (как в 0002), MP-table должна следовать.
- **Параметризовать число CPU.** Сейчас 1 CPU; для будущего SMP — нужно генерировать N processor entries.
- **Сохранить оригинальные байты ROM в полях, чтобы reset() мог восстановить overlay.** Сейчас, если `PC.reset()` снова прокатывает init, при повторном overlay захватится УЖЕ patched версия. Митигация: захватывать оригинал один раз.
- **Сделать `MpTable.BLOCK_BASE` опцией CLI** (`-mp-table-base`) — для отладки коллизий.
- **Bochs BIOS уже содержит `_MP_`/`PCMP` строки** в data templates. Стоит разобраться, рантайм код Bochs это публикует или нет — возможно, наш overlay избыточен.
- **`Bios.imageData` — `private`.** Если в будущем понадобится патчить imageData до load, добавить protected `patch(int, byte[])` метод.
