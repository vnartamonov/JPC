# Инкремент 0004 — B3: LAPIC + IO-APIC MMIO stubs

**Дата:** 2026-04-30
**Этап:** B (i386-совместимость с NetBSD), подэтап B3.
**Статус:** Реализация.
**Зависит от:** [0001.md](0001.md), [0002.md](0002.md), [0003.md](0003.md).

---

## 1. Цель

В 0003 мы декларировали через MP-table наличие LAPIC по адресу `0xFEE00000` и IO-APIC по `0xFEC00000`, но физически по этим адресам ничего не отвечает (`UnconnectedMemoryBlock` возвращает 0xFFFFFFFF и игнорирует записи). Современная гостевая ОС, прочитав MP-table, идёт по этим адресам — и в зависимости от политики либо panic'нет (на ICR-write «всегда 0xFFFFFFFF» она диагностирует «недопустимое состояние APIC»), либо tight-loop'нет в ожидании EOI/ID, либо тихо разойдётся с реальностью.

**Цель:** разместить **crash-proof MMIO storage** для обоих APIC. Регистры backed массивом `int[]`, корректные reset values (ID, version, LVT mask), dword-aligned R/W работают без `#GP`. **Реальной доставки прерываний нет** — этого достаточно, чтобы LAPIC init code path в NetBSD/Linux не упал; работающий timer и IPI-доставка — отдельный инкремент.

Минимальный инкремент: код + JUnit-тесты + зелёная сборка. **Без реального запуска NetBSD ISO.**

## 2. Местоположение и базовые свойства

| Устройство | База            | Размер блока | Регистры                                    |
|-----------|------------------|--------------|---------------------------------------------|
| LAPIC     | `0xFEE00000`     | 4 KB         | ~30 регистров в окне 0x000–0x3F0, dword-aligned |
| IO-APIC   | `0xFEC00000`     | 4 KB         | 2 прямых регистра: IOREGSEL (0x00), IOWIN (0x10) |

Оба ровно по `BLOCK_SIZE = 4096` — `PhysicalAddressSpace.mapMemory` примет.

## 3. LAPIC — регистры и reset values

Подмножество, важное для прохождения LAPIC-init в NetBSD/i386 GENERIC (см. `sys/arch/x86/x86/lapic.c`):

| Off   | Имя              | Reset       | Семантика в стабе |
|-------|------------------|-------------|---------------------|
| 0x020 | ID               | `0x00<<24`  | dword R/W (поле 24-31 — APIC ID; стаб хранит как полный dword) |
| 0x030 | VERSION          | `0x00050014`| RO (запись игнорируется); 6 LVT entries, P4-style |
| 0x080 | TPR              | 0           | R/W |
| 0x090 | APR              | 0           | RO storage |
| 0x0A0 | PPR              | 0           | RO storage |
| 0x0B0 | EOI              | —           | W-only, write игнорируется (no ISR semantics в стабе) |
| 0x0C0 | RRD              | 0           | RO storage |
| 0x0D0 | LDR              | `0xFF000000`| R/W |
| 0x0E0 | DFR              | `0xFFFFFFFF`| R/W |
| 0x0F0 | SVR              | `0x000000FF`| R/W |
| 0x100–0x170 | ISR (8 dw)  | 0           | RO storage |
| 0x180–0x1F0 | TMR (8 dw)  | 0           | RO storage |
| 0x200–0x270 | IRR (8 dw)  | 0           | RO storage |
| 0x280 | ESR              | 0           | R/W |
| 0x300 | ICR_LOW          | 0           | R/W (no IPI delivery) |
| 0x310 | ICR_HIGH         | 0           | R/W |
| 0x320 | LVT_TIMER        | `0x00010000`| R/W (mask=1 → masked) |
| 0x330 | LVT_THERMAL      | `0x00010000`| R/W |
| 0x340 | LVT_PERFCNT      | `0x00010000`| R/W |
| 0x350 | LVT_LINT0        | `0x00010000`| R/W |
| 0x360 | LVT_LINT1        | `0x00010000`| R/W |
| 0x370 | LVT_ERROR        | `0x00010000`| R/W |
| 0x380 | INITIAL_COUNT    | 0           | R/W |
| 0x390 | CURRENT_COUNT    | 0           | RO storage (никогда не декрементируется в стабе) |
| 0x3E0 | DIVIDE_CONFIG    | 0           | R/W |

Прочие смещения: dword-storage, инициализированные нулями (forward-compatibility).

### 3.1. Доступ

Реальный LAPIC требует 32-bit aligned 32-bit access. Стаб **толерантен**: `getDoubleWord/setDoubleWord` идут в `int[1024]`; `getByte/setByte` от `AbstractMemory` распаковывают через `getWordInBytes/setWordInBytes`. Для типового гостя достаточно dword-аксесса.

`executeReal/Protected/Virtual8086` бросают `IllegalStateException` (как `UnconnectedMemoryBlock`) — нельзя выполнять код в LAPIC-окне.

## 4. IO-APIC — регистры

| Off  | Имя       | Семантика                                               |
|------|-----------|---------------------------------------------------------|
| 0x00 | IOREGSEL  | dword R/W; младшие 8 бит — селектор индирект-регистра  |
| 0x10 | IOWIN     | dword R/W; пробрасывается на индирект-регистр по IOREGSEL |

Indirect registers (через IOREGSEL):

| Idx       | Имя                | Reset                         |
|-----------|--------------------|-------------------------------|
| 0x00      | IOAPICID           | `0x02 << 24` (ID=2 из MP-table 0003) |
| 0x01      | IOAPICVER          | `0x00170011` (24 entries, ver 0x11)  |
| 0x02      | IOAPICARB          | 0                             |
| 0x10–0x3F | IOREDTBL[0..23]·64 | masked (low dword bit 16=1)  |

Доступ к недокументированным indirect-индексам — read/write идёт в `int[64]` storage без эффекта, ничего не падает.

## 6. Тестовая стратегия

### 6.1. LAPIC

| Сценарий | Ожидание |
|---|---|
| `getSize()` | `4096` |
| Reset value: ID @0x020 | `0x00000000` |
| Reset value: Version @0x030 | `0x00050014` |
| Reset value: SVR @0x0F0 | `0xFF` |
| Reset value: LVT_TIMER @0x320 | `0x00010000` |
| Reset value: DFR @0x0E0 | `0xFFFFFFFF` |
| Reset value: LDR @0x0D0 | `0xFF000000` |
| R/W round-trip TPR @0x080 | written value читается обратно |
| R/W round-trip ICR_LOW @0x300 | written value читается обратно |

## 7. Что НЕ входит

- **Реальная доставка прерываний** — LVT_TIMER не тикает, ICR не отправляет IPI, ESR не накапливает ошибки.
- **EOI semantics** — write в EOI ничего не делает (в реальном LAPIC он сбрасывает старший установленный бит ISR).
- **TPR/PPR priority logic** — TPR хранится, PPR не пересчитывается.
- **IO-APIC IRQ routing** — IOREDTBL хранит конфигурацию, но реально IRQ от ISA-устройств всё ещё идут через 8259 (`InterruptController`), не через IO-APIC.
- **APIC global enable/disable** через `IA32_APIC_BASE_MSR` — мы регистрируем MMIO unconditionally; логика «выключения через MSR.bit11» отложена.
- **x2APIC** — не нужен.

## 8. Риски

- **Мап LAPIC поверх существующей памяти.** `0xFEE00000` далеко от обычной физпамяти (640 МБ), но выше границы 4 ГБ от 0 это уже область MMIO у реального PC. У JPC default RAM = 16 МБ ⇒ адрес гарантированно вне RAM. Безопасно.
- **Гость пишет в RO-регистр (Version) и ожидает изменения.** Спорный вопрос — реальный LAPIC игнорирует. Стаб тоже игнорирует ⇒ корректно.
- **Гость читает CCR, ожидая монотонного декремента.** Стаб всегда возвращает `INITIAL_COUNT` (точнее — то, что записано в storage CCR). Гость может зависнуть в polling-loop. Митигация: для B7 (boot ISO) потребуется доделать таймер до working-state.
- **MP-table из 0003 декларировала IO-APIC ID=2, но IOAPICID — это first indirect reg, он R/W.** Гость может прочитать ID=2 (наш default), может его перезаписать. Стаб не следит за консистентностью с MP-table. Реально NetBSD читает ID и пользуется им.
- **Порядок init.** Аналогично 0003, `LocalApic`/`IoApic` ждут `PhysicalAddressSpace`. Должны успеть зарегистрировать MMIO раньше любых обращений к этим адресам гостем — а гость не запустится, пока configure() не закончится. Безопасно.

## 10. Предложения по улучшению

- **Унификация MMIO-устройств.** LAPIC и IO-APIC структурно похожи; стоит выделить `MmioMemory extends AbstractMemory` с общим backing-`int[]` + dictionary reset values.
- **Save/load state.** Сейчас `AbstractHardwareComponent.saveState/loadState` пустые ⇒ снапшот не сохранит состояние APIC (важно для time-travel). Кандидат на доработку.
- **`InterruptController` (8259) не интегрирован с LAPIC LINT0.** Реально LINT0 принимает ExtINT от 8259. В стабе LINT0 — просто хранилище. Для B7 это — следующая логическая работа.
- **`int[]` хранение для LAPIC расходует 4 КБ на устройство.** Это нормально, но для time-travel snapshot — не идеально. Можно сжать до actually-used ~30 регистров, но в стабе делаем «всё mapping → массив».
- **`SYS_RAM_SIZE` static** (см. 0001 §7) — мешает мульти-инстансу APIC если будут несколько `PC` в одной JVM. Сегодня это и так не работает.
