**Дата:** 2026-04-30
**Этап:** B (i386-совместимость с NetBSD), подэтап B6.
**Статус:** Реализация.
**Зависит от:** [0001.md](0001.md).

---

## 1. Цель

NetBSD драйвер `wd(4)` (ATA disk) и `cd(4)` (ATAPI CD-ROM) подаёт на `IDEChannel` поток команд во время install: IDENTIFY, READ DMA, WRITE DMA, FLUSH CACHE для дисков; ATAPI PACKET → MODE SENSE / READ TOC / READ 10 для CD-ROM. Если что-то не так (таймаут, неподдерживаемая команда, нарушение протокола), с точки зрения JPC пользователь видит «гость завис», и без инструментации — невозможно сказать, какая именно команда вызвала проблему.

**Цель:** добавить лёгкую инструментацию `IDEChannel`:
- Атомарные счётчики команд по opcode (ATA + ATAPI отдельно).
- Snapshot-снимок текущих счётчиков для диагностики.
- Mnemonic-имена для известных опкодов (для удобства отчёта).
- CLI-флаг `-trace-ide` — когда установлен, каждая команда дополнительно логируется через `java.util.logging` на уровне `INFO`.

Минимальный инкремент: код + JUnit-тесты + зелёная сборка. **Без реального запуска NetBSD ISO** — проверка работы в гостевой ОС оставлена на B7.

## 3. Дизайн `IdeStats`

```java
public final class IdeStats {
    public void recordAta(int opcode);
    public void recordAtapi(int opcode);

    public long getAtaCount(int opcode);
    public long getAtapiCount(int opcode);
    public long getTotalAtaCommands();
    public long getTotalAtapiCommands();

    public Map<Integer, Long> ataSnapshot();   // только ненулевые
    public Map<Integer, Long> atapiSnapshot(); // только ненулевые

    public static String ataMnemonic(int opcode);   // "WIN_READDMA" или "0x21"
    public static String atapiMnemonic(int opcode); // "GPCMD_READ_10" или "0x55"

    public void reset();
}
```

- Хранилище: два `long[256]` (по индексу opcode). Никаких HashMap в hot path — это I/O-канал, обращений много.
- Атомарность: на single-threaded JPC writeIDE из одного потока, инкремент через `++` достаточен. Но для будущей thread-safety используем `AtomicLongArray` — overhead минимален.
- Mnemonic table — статический `Map<Integer,String>`, заполняется через рефлексию из `IDEChannel.IDEState` (поля `WIN_*` и `GPCMD_*`) или вручную. **Ручной вариант** надёжнее — рефлексия по private inner-class сложна.

## 4. Точки врезки в IDEChannel

### 4.1. ATA: writeIDE case 7

```java
case 7:
    if (currentDevice != devices[0] && currentDevice.drive == null) break;
    stats.recordAta(data & 0xFF);
    if (Option.trace_ide.isSet())
        LOGGING.log(Level.INFO, "ATA cmd 0x{0} ({1})",
            new Object[]{Integer.toHexString(data & 0xFF), IdeStats.ataMnemonic(data & 0xFF)});
    switch (data) { ... }
```

### 4.2. ATAPI: IDEState.atapiCommand()

```java
private void atapiCommand() {
    int op = 0xff & ioBuffer[0];
    stats.recordAtapi(op);
    if (Option.trace_ide.isSet())
        LOGGING.log(Level.INFO, "ATAPI cmd 0x{0} ({1})",
            new Object[]{Integer.toHexString(op), IdeStats.atapiMnemonic(op)});
    switch (op) { ... }
}
```

`IdeStats stats` хранится в `IDEChannel` — он же создаёт `IDEState` инстансы внутри. Доступ изнутри `IDEState.atapiCommand()` — через outer reference (`IDEChannel.this.stats`).

Проверка флага через `Option.trace_ide.isSet()` происходит на каждой команде. Это IO-канал, не CPU hot path; overhead приемлем.

## 6. Что НЕ входит

- **Per-IDE-channel breakdown.** Сейчас счётчики per-IDEChannel (PIIX3 имеет 2 канала); агрегация — на стороне consumer'а.
- **Latency / timing.** Только counts. Latency-распределения отдельный инкремент.
- **Anomaly detection.** Не детектируем «команда при BUSY=1». Оставлено на дальнейшую инкрементацию.
- **Exposed через IO-port для гостя.** Stats доступен только из Java-кода (тесты, debugger). Гость их не видит.
- **Save/load state.** Счётчики обнуляются при snapshot reload — нет смысла хранить в save state.

## 7. Риски

- **Mnemonic-таблица отстаёт от констант в `IDEState`.** Ручная синхронизация. Митигация: тесты дёргают именно те значения, что в `IDEState.WIN_*`/`GPCMD_*` — рассинхронизация поймается компилятором.
- **`AtomicLongArray` overhead** — обычно <5% на op в JVM, для I/O channel приемлем.
- **`Option.trace_ide.isSet()` в hot path** — два static-метод вызова на каждую команду. Минимально.
- **Логирование через `java.util.logging`** — формат `Level.INFO` с placeholder. Если у пользователя глобально INFO выключен — лог теряется. Документировать в help-строке.

## 9. Предложения по улучшению

- **Унифицировать stats-инфраструктуру.** Аналогичные счётчики (по портам, по DMA-каналам, по PCI-конфигурационным циклам) могут потребоваться позже. Кандидат на общий `Counters` namespace.
- **Per-device counters.** Сейчас `IdeStats` per-channel; внутри канала смешан `master`/`slave`. Разделение полезно для CD-ROM vs HDD.
- **CSV-dump на shutdown.** Опция `-trace-ide-out file.csv` — выгружать snapshot при `PC.stop()`. Удобно для post-mortem анализа долгих boot-сессий.
- **Mnemonic generator.** Текущий ручной список — кандидат на codegen из `IDEState` через annotation processor или просто sed-скрипт.
