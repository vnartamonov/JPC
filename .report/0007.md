# Инкремент 0007 — B7: попытка boot NetBSD/i386 11.0-RC2

**Дата:** 2026-04-30
**Этап:** B (i386-совместимость с NetBSD), подэтап B7 — приёмочный.
**Статус:** Эксперимент проведён. Boot **значительно** превзошёл ожидания. Документирую результаты и следующие шаги.
**Зависит от:** [0001.md](0001.md) ... [0006.md](0006.md) — все наработки этапа B.

## 1. Что делали

Скачали `boot-com.iso` (NetBSD/i386 11.0_RC2, 253 МБ, с serial-консолью) с `cdn.netbsd.org`. Запустили эмулятор:

```
java -jar target/jpc-app/JPCApplication.jar \
     -boot cdrom -cdrom /tmp/netbsd-bootcom.iso \
     -ram 64
```

Headless-флаг `-no-screen` декларирован в `Option.java`, но в `JPCApplication.main()` НЕ обрабатывается — frame создаётся всегда. На macOS Swing-окно открывается без проблем; в `-Djava.awt.headless=true` падает на `JFrame` constructor (это known gap, см. §4 предложений).

Параллельно второй прогон с `-trace-ide` для логирования IDE-команд. Тайм-аут 60-180 секунд каждый прогон (kill -9 PID).

Фатальный trap:
```
[   1.0098273] fatal privileged instruction fault in supervisor mode
[   1.0098273] trap type 0 code 0 eip 0xc0102837 cs 0x8 eflags 0x10206 cr2 0 ilevel 0x8 esp 0
[   1.0098273] curlwp 0xc1515100 pid 0 lid 0 lowest kstack 0xc18c72c0
[      kernel: supervisor trap privileged instruction fault, code=0
[      Stopped in pid 0.0 (system) at  netbsd:spllower+0x37:   addb    %al,0(%eax)
```

`esp 0` означает стек безнадёжно повреждён; падение в `spllower` (Set Priority Level Lower) — функция, которая на NetBSD/i386 c LAPIC обращается к LAPIC TPR.

## 3. Анализ найденных проблем

### 3.1. ACPI/MP-overlay конфликтуют с Bochs BIOS dynamic tables (P1)

Bochs BIOS на старте rombios32 печатает:
```
bios_table_addr: 0x000fb778 end=0x000fcc00
MP table addr=0x000fb850 MPC table addr=0x000fb780 size=0xd0
SMBIOS table addr=0x000fb860
```

Bochs BIOS **сам генерирует** MP-table и SMBIOS в области 0xfb778-0xfcc00, поверх наших EPROM-overlays 0xFB000 (ACPI из 0005) и 0xFC000 (MP-table из 0003). Поскольку наши overlays — **read-only EPROM**, записи Bochs BIOS теряются. NetBSD затем не находит наш RSDP («A valid RSDP was not found») и Bochs-овский (потому что кадровые байты по итогу — мусор).

Что делать:
- **Вариант A**: убрать наши `MpTable`/`Acpi` оверлеи. Положиться на Bochs BIOS, который и так строит MP-table и (в свежих версиях) ACPI таблицы.
- **Вариант B**: переместить наши оверлеи в другую область (например, 0xE0000-0xEFFFF — пустой нижний BIOS-сегмент).
- **Вариант C**: сделать наши оверлеи writable (Shadow-RAM-style) и предоставить таблицы как «начальное содержимое», которое Bochs может перезаписать.

Рекомендую **Вариант A** — сократить объём нашего кода, доверить Bochs BIOS.

### 3.2. CPUID в ядре показывает Pentium (id 0x513), не PII (0x634) (P2)

Мы выставили default `cpuLevel=6` в 0002, но ядро NetBSD рапортует `id 0x513` (Pentium). Гипотезы:
- В коде где-то ещё используется hardcoded или вычисленный `cpulevel != 6`.
- Bochs BIOS перезаписывает CPUID-сигнатуру на этапе инициализации.
- Один из путей CPUID (вход с другим EAX) не различает 5 vs 6.

Стоит добавить трассировку CPUID-вызовов и проверить.

### 3.3. Падение в `spllower+0x37` (P0, blocker для дальнейшего boot)

Кандидаты:
1. **`cmpxchg8b` без CPUID-bit CX8.** PII featureset в `Processor.cpuid` (level 6) не включает bit 8 (CX8). NetBSD i386 ABI использует cmpxchg8b в SPL fast path. Если JPC вообще не реализует cmpxchg8b → #UD.
2. **LAPIC TPR-write путь.** spllower на NetBSD/i386 с LAPIC (а наша 0004 LAPIC stub доступен) пишет в TPR (offset 0x80). Затем читает PPR (0xA0) для проверки приоритета. Наш стаб TPR — R/W, PPR — RO в 0. Кернел может попытаться диспетчеризировать pending interrupt по PPR=0, прыгнуть в обработчик, который не настроен.
3. **Какой-то privileged opcode** в spllower, не реализованный JPC. Например, `monitor`/`mwait` (Intel idle hints) — но это не privileged.

**Самое правдоподобное:** комбинация 3.2 + 3.3.1: ядро видит CPU как Pentium (586), но всё равно эмитит `cmpxchg8b` (compiled with -mtune=i486 minimum, CX8 mandatory с Pentium). JPC может его не поддерживать → #UD.

### 3.4. `int13_cdrom: unsupported AH=08` (P3, не блокер)

Bochs BIOS не реализует int13 AH=08 (Get Drive Parameters) для CD-ROM. NetBSD bootloader пробует, получает CF=1, fallback на другой API (вероятно AH=42 LBA read). Кернел загружается несмотря на это.

### 3.5. `IDE time out` в Bochs BIOS init (P3)

При перечислении IDE-устройств Bochs BIOS не дождался какого-то ответа от первичного канала. Не блокирует boot. Возможно связано с тем, что `master` нет (только `slave` CDROM).

### 3.6. `-trace-ide` flag не вывел ATA/ATAPI команды (P2 для 0006)

При запуске с `-trace-ide` ни один `ATA cmd ...` или `ATAPI cmd ...` не появился в логе, хотя ясно происходила activity (BIOS читает kernel с CDROM, kernel в device probe). Возможные причины:
- Логгер `IDEChannel` имеет отдельный logger, INFO не пропагируется до root.
- Хук в `writeIDE` не достигается (другой code path).
- `Option.trace_ide.isSet()` false по какой-то причине.

Нужно дебажить отдельно.

### 3.7. `-no-screen` flag декларирован, но не обрабатывается (P3)

`Option.noScreen` создаётся в `Option.java:65`, но `JPCApplication.main()` всегда инстанцирует `JPCApplication` с JFrame. Под `java.awt.headless=true` падает с HeadlessException. Простой fix: проверить `Option.noScreen.isSet()` перед `setVisible(true)` и пропустить frame, или иметь альтернативный bare main без UI.

## 4. Минимальные следующие шаги к sysinst

В порядке приоритета:

1. **0008 (рекомендуется)**: Реализовать `cmpxchg8b` в JPC (если ещё нет). Проверить через дизассемблирование `eip 0xc0102837` — при возможности exec NetBSD ядра в JPC с trace-инструкций.
2. **0009**: Убрать `MpTable`/`Acpi` overlays (или переместить в 0xE0000-0xEFFFF). Положиться на Bochs BIOS.
3. **0010**: Починить `Option.trace_ide` логирование. Возможно через `Logger.getLogger("").setLevel(Level.INFO)` или использовать `System.err.println`.
4. **0011**: `-no-screen` flag действительно делать headless — пропускать `JPCApplication` JFrame.
5. **0012 (опционально)**: Bumb default `Option.ram` 16→64 МБ — иначе пользователь должен помнить о `-ram 64`.
6. **0013 (большой)**: Working LAPIC timer — необходим для запущенного userland NetBSD (после spllower-фикса всё равно потребуется).

- JPC's 32-битная PC-эмуляция качественная и достаточная для NetBSD/i386 bootloader + ранние стадии kernel.
- Главный оставшийся блокер — **минимальная LAPIC SPL**-функциональность и отсутствие cmpxchg8b/иных инструкций. Эти проблемы хорошо понятны, локализуемы.

## 7. Предложения по улучшению

- **Test-fixture для NetBSD smoke-тест.** Сохранить boot-com.iso в `tests/fixtures/` и иметь автоматизированный smoke-test «kernel banner появился за 60 секунд». Сейчас полностью ручной.
- **Логирование serial port в стандартизированный формат.** java.util.logging форматирование добавляет много noise (timestamp, class). Для serial console читаемее plain text. Кандидат: `-Djava.util.logging.SimpleFormatter.format=%5$s%6$s%n` или custom handler.
- **`-no-screen` headless** — должен работать. Сейчас задекларирован, не используется. Кандидат на 0010.
- **Отдельный exit-flag** для эмулятора, чтобы не приходилось `kill -9`. Сейчас единственный путь завершения — закрыть Swing окно, которого в headless нет.
- **Экспорт snapshot-метрик при exit** — в 0006 есть IdeStats; на shutdown полезно дампить counters. Связано с 0006 §9 «CSV-dump on shutdown».
- **`spllower+0x37` нужно дизассемблировать** для понимания, какая инструкция упала. Использовать `tools.Tools` или `objdump` на kernel image из ISO. Это превратит «гипотезу cmpxchg8b» в факт.
