# Инкремент 0008 — B7-followup: продвинуть boot за `spllower+0x37`

**Дата:** 2026-04-30
**Статус:** Реализация (P0 из 0007).
**Зависит от:** [0007.md](0007.md).

---

## 1. Цель

В 0007 NetBSD/i386 11.0 прошёл bootloader, kernel banner, полный device probe и упал в `spllower+0x37` с «privileged instruction fault, esp 0». Цель этого инкремента — **локализовать конкретную инструкцию**, реализовать либо обойти её, и подтвердить, что boot продвинулся дальше этой точки.

Минимальный инкремент: код + JUnit + зелёная сборка + успешное прохождение spllower (kernel доходит до новой точки или sysinst).

## 2. План

1. **Использовать infrastructure ISO.** В `infrastructure/NetBSD-11.0_RC2-i386.iso` (~710 МБ, full install). Не качать заново.
2. **Запустить JPC с `-log-disam` и `-log-disam-addresses`** — redirect stdout/stderr в файл.
3. **Найти блок, содержащий `eip=0xc0102837`** в логе — `LOG_DISAM_ADDRESSES` печатает `Disassembling PM from <addr>` при первом decode. `PRINT_DISAM` затем печатает class name каждой инструкции в блоке. Сопоставление даст имя класса инструкции на смещении 0x37 от блока.
4. **Идентифицировать инструкцию.** Кандидаты:
   - `cmpxchg8b` — JPC реализован, но возможно CX8 не объявлен в CPUID, и кернел получает trap при NetBSD-side проверке.
   - LAPIC TPR/EOI MMIO в стабе из 0004 — недостаточная семантика.
   - Privileged instruction вроде `int 0x82` (NetBSD `Xspllower` callgate?).
5. **Исправить.** Объём правки зависит от находки. Кандидаты:
   - Добавить CX8 в CPUID feature flags для `cpuLevel >= 5`.
   - Расширить LAPIC stub до возврата non-zero PPR при write-TPR.
   - Реализовать missing instruction в opcodes/.
6. **Перезапустить.** Подтвердить новое поведение (либо новая точка падения, либо успех).
7. **Тесты.** Покрыть фикс (CPUID bit, LAPIC behaviour, opcode behavior).

## 3. Ожидания и риски

- **Скорее всего CX8 (cmpxchg8b feature bit).** В Processor.cpuid level 6 features: PAE(6), SYSENTER(11), CMOV(15), MMX(23). Bit 8 (CX8) не объявлен. NetBSD i386 ABI требует CMPXCHG8B как baseline (Pentium+). Если NetBSD проверяет CPUID bit 8 и видит 0 — она избегает cmpxchg8b. Если же она использует напрямую (без проверки), JPC её исполняет (в коде есть `cmpxchg8b_M.java`), и логика теоретически работает — но есть ли там семантический баг?
- **Альтернатива: NetBSD читает что-то от LAPIC.** В spllower+0x37 может быть MMIO read/write в LAPIC-окно. Стаб из 0004 возвращает фиксированные значения, кернел может неправильно их интерпретировать.
- **Третий кандидат: NetBSD выполняет инструкцию которой JPC не имеет.** Вряд ли — `spllower` маленькая функция базовых операций.

Если за разумное время не удаётся локализовать — выписать в отчёт, что log-disam недостаточен, и закрыть инкремент с поправкой плана на следующий.

  ```
  java.lang.NullPointerException: Cannot invoke
      "org.jpc.emulator.memory.Memory.getDoubleWord(int)" because "m" is null
      at org.jpc.emulator.memory.LinearAddressSpace.getDoubleWord(LinearAddressSpace.java:592)
  ```
  То есть `validateTLBEntryRead(offset)` вернул `null`, что не должно было быть.

### 7.2. Уточнённая гипотеза root cause (статический анализ кода)

Корневая причина — **`PhysicalAddressSpace.getMemoryBlockAt(int)` может вернуть `null`** для физических адресов, у которых `index[topIdx]` уже выделен (т.к. соседний блок был замаплен), но конкретный `index[topIdx][botIdx]` — `null` (новый Java-массив `new Memory[BOTTOM_INDEX_SIZE]` заполняется `null`-ами).

```java
// PhysicalAddressSpace.java:858-868
private Memory getMemoryBlockAt(int i) {
    try {
        return quickIndex[i >>> INDEX_SHIFT];
    } catch (ArrayIndexOutOfBoundsException e) {
        try {
            return index[i >>> TOP_INDEX_SHIFT][(i >>> BOTTOM_INDEX_SHIFT) & BOTTOM_INDEX_MASK];
            // ↑ ЕСЛИ массив index[a] выделен, но элемент [b] == null,
            //   обычный array access возвращает null без NullPointerException
        } catch (NullPointerException n) {
            return UNCONNECTED;
        }
    }
}
```

Java-семантика: `array[i]` возвращает `null`, если элемент null — NPE НЕ бросается. NPE catch здесь срабатывает только когда сам `index[a]` (внешний массив) равен null.

Этот null проникает в TLB через `tlb.setReadMemoryBlockAt(..., null)` в `LinearAddressSpace.validateTLBEntryRead` (line 343). Затем `tlb.getReadMemoryBlockAt(...)` возвращает этот null обратно. Следующее обращение `m.getDoubleWord(...)` бросает NPE.

`new Memory[BOTTOM_INDEX_SIZE]` создаётся в `setMemoryBlockAt` при первом mappping в данную TOP-row. После этого всё, что не вызывало `setMemoryBlockAt`, остаётся null.

Когда NetBSD после device probe начинает реальную работу с памятью, оно ходит по виртуальным адресам, page table walk даёт физический адрес, который попадает в неинициализированный слот → null → hang.

### 7.3. Скорректированный план

1. **Добавить диагностический CLI-flag `-log-tlb-null`.** На каждое возвращение null из `PhysicalAddressSpace.getMemoryBlockAt` и из `LinearAddressSpace.validateTLBEntry{Read,Write}` — печатать линейный/физический адрес, путь (4K vs 4M, paging on/off), CR3, isSupervisor. Этот лог подтвердит гипотезу до фикса.
2. **Запустить boot c `-log-tlb-null`.** Убедиться, что null приходит именно из `PhysicalAddressSpace`. Зафиксировать, какие физические адреса проблемные.
3. **Фикс:** в `PhysicalAddressSpace.getMemoryBlockAt` вернуть `UNCONNECTED`, если итоговый элемент массива равен null. Это семантически корректно: «не замапленный физический адрес ⇒ floating high (`-1`)».
4. **JUnit-тесты:**
   - `PhysicalAddressSpaceTest`: вызвать `getByte`/`getDoubleWord` для физического адреса, лежащего в TOP-row, где есть хотя бы один маппинг рядом, но конкретный block не замаплен — должно вернуть -1, не null/NPE.
   - `LinearAddressSpaceTest`: проверить, что paging walk через unmapped физическую память не зависает (через linear address space).
5. **Перезапустить boot** на `boot-com.iso` — kernel должен пройти текущую точку C01279B1.
6. **Обновить отчёт** + добавить запись в `MAIN.md` для 0008.

### 7.5. Подтверждение runtime-логом

Прогон `boot-com.iso` с `-log-tlb-null -eip-poll 5000 -ram 64`, тайм-аут 200 секунд:

```
[tlb-null] PhysicalAddressSpace.getMemoryBlockAt: index[0x3fb][0x140] is null, phys=0xfed40000 (#1)
```

`top_idx=0x3FB`, `bot_idx=0x140` ⇒ phys = `(0x3FB << 22) | (0x140 << 12)` = `0xFED40000`.

Это адрес в окне HPET / extended LAPIC region. TOP-row `0x3FB` уже была выделена ранее в `setMemoryBlockAt` для нашего LAPIC stub-а на `0xFEE00000` (инкремент 0004), но конкретный 4 КБ слот `0xFED40000` остался `null`. NetBSD во время probe HPET-а выполнял `mov reg32, [linear-addr]`, page table walk резолвил его в этот phys, `target.getReadMemoryBlockAt` возвращал null, который проникал в TLB и затем дёргал NPE в `mov_Gd_Ed_mem.execute`.

После фикса:
- 1 hit `[tlb-null]` (тот же физ-адрес, ожидаемое поведение, не сбой)
- **0 STALLED меток** — `vmClock.getTicks()` непрерывно растёт
- **0 NullPointerException** в `/tmp/jpc.stdout`/`/tmp/jpc.stderr`
- Kernel banner `NetBSD 11.0_RC2 (GENERIC) #0` напечатан
- Достигнута точка `spllower+0x37` privileged-instruction fault — **тот же next blocker, что в 0007**

То есть NPE-hang из 0008 closed; следующая преграда (spllower) — отдельный issue, описанный в 0007 §3.3.
