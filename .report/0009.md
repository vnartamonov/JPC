# Инкремент 0009 — флаги -no-mp / -no-acpi; kernel banner + device probe

**Дата:** 2026-05-03
**Статус:** Готов
**Зависит от:** [0008.md](0008.md)

---

## 1. Цель

Добавить флаги `-no-mp` и `-no-acpi`, которые отключают внедрение MP-таблицы/LAPIC/IO-APIC/ACPI overlay'ев в PC. По итогам 0008 диагностика показала, что NetBSD видит наш MP-table, запускает `lapic_init()`, устанавливает CPUF_APIC, применяет `alternatives_apply()` — и часть альтернативного кода ломается (spllower+0x37 попадает в #UD). Гипотеза: без MP-таблицы NetBSD уйдёт на PIC-путь, где spllower проще.

## 2. Реализация

### 2.1. Новые флаги (Option.java)

public static final Switch no_mp   = createSwitch("no-mp");
public static final Switch no_acpi = createSwitch("no-acpi");

## 5. Анализ нового блокера: spllower+0x37

### 5.1. Контекст

`spllower` — функция управления уровнем прерываний в NetBSD/i386. С поддержкой APIC в GENERIC-ядре используется ALTERNATIVE-механизм:
- **Default (без APIC):** `00 00` — placeholder (`addb %al,0(%eax)`)
- **Patched (с APIC):** запись в LAPIC TPR через `lapic_base + LAPIC_TPRI`

Патчинг происходит в `alternatives_apply()` при старте, если ядро обнаруживает APIC через CPUID bit 9 (`CPUF_APIC`) или MP-таблицу.

С флагом `-no-mp`:
- NetBSD не видит MP-таблицу → `lapic_init()` не вызывается
- Но ALTERNATIVE placeholder `00 00` НЕ патчится
- При вызове `spllower(0)` с eax=0 выполняется `addb %al,0(%eax)` = запись в виртуальный адрес 0

### 5.2. Почему #UD (vector 6) вместо #PF (vector 14)?

FaultLogger фиксирует `type=UNDEFINED vector=0x06`. Инструкция `addb %al,0(%eax)` — корректная x86, и должна вызывать #PF (если адрес 0 не маппирован). Две возможные причины:

a) **Адрес 0 маппирован** (временный identity-map ядра или BIOS IVT). Инструкция выполняется, но последующий код в spllower содержит что-то, вызывающее #UD (кандидат: `ltr` или `sldt` в ring0-sensitive функции). FaultLogger показывает байты в момент fault, которые уже обновлены kernel-patching'ом.

b) **Stale JIT block.** До patching ядро записало в slot C0102837 `ud2` (0F 0B) как trap-guard. JPC invalidated block (regionAltered работает), но re-decode проходит после ALTERNATIVE has written `00 00`. Однако #UD уже был dispatch до re-decode... этот путь требует дополнительной проверки.

В любом случае: NetBSD DDB вызван, ядро живо, ticksDelta > 0 — прогресс значительный.

### 5.3. Предложения по исправлению (для 0010)

1. **Включить CPUID bit 9 (APIC)** в `Processor.java` → `alternatives_apply()` запатчит spllower на LAPIC-путь. Работает при условии что LocalApic stub корректно обрабатывает TPR write.
2. **Убрать `-no-mp`** и включить CPUID bit 9 → `lapic_init()` будет вызван, `lapic_base` получит адрес нашего stub, spllower запишет в LAPIC TPR.
3. **Диагностика:** добавить флаг `-log-ud` который при получении #UD дампит дополнительный контекст (содержимое IDT[6], тип gate, выполнение `followProtectedModeException`).

## 6. Предложения по улучшению

- **CPUID bit 9 (APIC) следует включить в следующем инкременте** — текущий дефолт (закомментировано в Processor.java строка 1303/1359) блокирует патчинг ALTERNATIVE в spllower.
- **Добавить тест boot с `-no-mp -no-acpi`** как regression проверку (ожидаем kernel banner в stderr за 200 секунд).
- **`-no-mp` логичнее переименовать** в `-no-smp` (это standard Linux/BSDs convention), но breaking change — оставить как есть.
