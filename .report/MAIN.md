# JPC — Журнал инкрементов

Индекс работ по доработке эмулятора JPC. Каждый инкремент — отдельный файл `NNNN.md` рядом.

## Принципы

- Один инкремент = одна минимальная задача с критерием «фича добавлена + тест + сборка зелёная».
- В каждом файле инкремента есть блок **«Предложения по улучшению»** — туда пишем замечания, которые выходят за рамки текущего инкремента.
- Этот файл (`MAIN.md`) — только оглавление и сквозные принципы. Он не содержит самого контента инкрементов.

## Текущий target ОС

NetBSD 11.0-RC2. По итогам инкремента 0001 **рекомендуется** переключить target с amd64 на i386 — обоснование в [0001.md §5](0001.md). Решение остаётся за владельцем проекта.

## Сквозные блокеры (известные)

- Отсутствие long mode в JPC ⇒ NetBSD/amd64 заблокирован до Этапа C (см. 0001.md §4).
- LAPIC/IO-APIC — crash-proof MMIO storage есть (0004), но прерывания не доставляются и таймер не тикает. Требуется для userland.
- ACPI 1.0 (0005) и MP-table (0003) — overlay'ы конфликтуют с Bochs BIOS dynamic tables в той же области (0xfb778-0xfcc00). NetBSD не находит RSDP. См. 0007.md §3.1.
- **NetBSD/i386 kernel падает на spllower+0x37** (eip=C0102837): байты `00 00` — ALTERNATIVE-placeholder не патчится, т.к. CPUID bit 9 (APIC) закомментирован в Processor.java. Починка: раскомментировать `features |= (1<<9)` и убрать `-no-mp`. С MP-таблицей `lapic_init()` ставит `lapic_base`=0xFEE00000; `alternatives_apply()` патчит spllower на TPR-write в наш LocalApic stub. **Следующий инкремент: 0010.** См. 0009.md §5.
- Default RAM = 16 МБ — мал для NetBSD (нужно ≥64 МБ; bootloader работает только с явным `-ram 64`).
- `-no-screen` декларирован, не реализован. `-trace-ide` логирование не доходит до root logger.

## Сквозные предложения по улучшению

Каждый инкремент дописывает свои в свою секцию «Предложения». Здесь — только консолидированные сквозные пункты, которые повторяются.

- **Унифицировать чтение `cpulevel`** — повторно встречается в 0002.md (Processor.java читает значение в двух местах с одним и тем же default).
- **`cpuLevel`/`SYS_RAM_SIZE` как `static`** — мешают мульти-инстансной эмуляции; кандидат на инстанс-поля.
- **Дрейф help-текста в `Option.printHelp()`** — поправляется вручную в каждом инкременте; кандидат на автогенерацию.

### Предложения от 0008 (TLB null-safety)

- **Добавить help-строку для `-log-tlb-null`** в `Option.printHelp()` — флаг работает, но в `-help` не упомянут.
- **Аналогичная защита от `null` в `LinearAddressSpace.validateTLBEntryRead/Write`** — после фикса `PhysicalAddressSpace` теоретически в TLB null уже не попадёт, но defensive-нayar (если result == null → return UNCONNECTED + log) уберёт целый класс будущих регрессий. Сейчас цена `m == null` в callerах — NPE → hang; цена defensive-проверки — несколько ns. Стоит добавить в следующем инкременте.
- **Превратить `UNCONNECTED` в видимый в дебаггере объект** — сейчас это безымянный `private static final Memory UNCONNECTED`, при наличии множества unmapped read'ов он практически невидим. Полезно при отладке: префиксовать `toString()` чем-то более диагностически ценным (напр. строкой адреса первого hit-а).
- **Симметричный `getMemoryBlockAt` контракт в Javadoc** — сейчас public-абстрактные методы `getReadMemoryBlockAt`/`getWriteMemoryBlockAt` не документируют, что null недопустим. После фикса 0008 — это формально часть контракта; стоит зафиксировать в Javadoc, чтобы не пришлось чинить заново.
- **Регресс-смок-тест на boot NetBSD `boot-com.iso`** — сейчас «boot до spllower» подтверждается только вручную. Кандидат: интеграционный тест в `tests/`, использующий тот же `boot-com.iso` как fixture, проверяющий что за N секунд в stderr появилась строка `NetBSD 11.0_RC2 (GENERIC)`. Запускается отдельным профилем Maven (`-Pintegration`), чтобы не замедлять основной CI.
- **Cap-логика TlbNullLogger** дублирует паттерн «throttle + summary», который пригодился бы и в `IdeStats` (0006), `LocalApic` (0004), и др. Кандидат: общий util `ThrottledLogger` в `org.jpc.support`.
